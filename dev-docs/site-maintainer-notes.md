# Site Maintainer Notes

## How this site was created

The site was scaffolded as an Astro project using the Starlight template and then adapted in place for Retro Systems Lab.

Core setup:

- framework: Astro
- docs theme: Starlight
- package manager: npm
- runtime used during setup: Node 22
- hosting target: GitHub Pages

Working commands:

```bash
npm install
npm run dev
npm run build
npm run preview
```

Main files to know:

- `astro.config.mjs`: site title, description, GitHub Pages base path handling, and sidebar structure
- `src/content/docs/index.mdx`: homepage content
- `src/content/docs/`: public site content
- `.github/workflows/deploy.yml`: GitHub Pages build and deploy pipeline

## Decisions and why they were made

### 1. Astro + Starlight

This was chosen because the project is primarily documentation and reference material, not a custom web app.

Why:

- content is written as Markdown or MDX
- the site stays static and low-maintenance
- built-in docs navigation fits the project better than a blank Astro app
- search comes from Starlight/Pagefind without needing a hosted search service

### 2. GitHub Pages deployment

The site is deployed with GitHub Actions to GitHub Pages.

Why:

- free hosting
- simple workflow for a mostly static documentation site
- no server maintenance
- easy fit for a repo-driven project

### 3. Dynamic base path for GitHub Pages

`astro.config.mjs` derives `site` and `base` from GitHub Actions environment variables during CI builds.

Why:

- local development stays simple at `/`
- deployment works even if the local folder name and GitHub repo name differ
- avoids hardcoding the repo name in config

### 4. Manual sidebar, not autogenerated navigation

The sidebar is defined manually right now.

Why:

- the project is still small
- the maintainer can keep the public information architecture intentional
- early pages can act as anchors for future content categories

If the content volume grows substantially, switching some sections to autogenerated navigation may become worthwhile.

### 5. Maintainer docs live outside `src/content/docs/`

Maintainer notes are stored in `dev-docs/`.

Why:

- avoids accidental publication
- keeps internal notes separate from public-facing content
- makes it clear which files are part of the public site build

## What the current structure is designed for

The public structure is built around different kinds of knowledge the project is expected to collect over time.

### `docs/`

Use this for reproducible instructions and setup notes.

Examples:

- installing SDKs
- building a MIDlet
- getting an emulator running
- documenting exact steps that worked

This section answers: **how do I do this again?**

### `tools/`

Use this for toolchain components, emulators, utilities, and environment notes.

Examples:

- emulator comparisons
- build tools
- inspection tools
- compatibility notes

This section answers: **what do I need to use?**

### `archives/`

Use this for preserved artifacts and records of historical material.

Examples:

- SDK installers
- version lists
- release notes
- device profiles
- recovered manuals

This section answers: **what survives, and what exactly is it?**

### `stories/`

Use this for historical context, explanations, retrospectives, and narrative material.

Examples:

- platform histories
- vendor differences
- distribution context
- why certain technical constraints mattered

This section answers: **why did the ecosystem work this way?**

### Homepage + seed pages

The homepage is intentionally brief and directional. It should stay focused on what Retro Systems Lab explores and where readers should go next.

The current starter pages are seed content, not a final taxonomy. They exist to establish the shape of the site and give each section a visible starting point.

## Future direction this structure supports

This structure is meant to support a growing archive of retro computing and mobile platform research without forcing everything into a single “blog” format.

It should work well if the project grows into:

- step-by-step platform bring-up notes
- long-lived reference pages
- curated software and SDK archives
- historical essays tied back to specific technical artifacts

The intended pattern is:

1. experiments start as notes
2. repeatable findings become `docs/`
3. reusable environment information goes into `tools/`
4. preserved artifacts go into `archives/`
5. context and interpretation go into `stories/`

## Maintenance guidance

- Keep anything public under `src/content/docs/`.
- Keep anything maintainer-only under `dev-docs/`.
- Prefer updating existing reference pages over creating many thin pages.
- Treat `archives/` as factual inventory and `stories/` as interpretation.
- Keep the homepage curated rather than exhaustive.
- Revisit the manual sidebar once the site has enough pages that hand-curation becomes friction.
